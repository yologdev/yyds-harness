Why this matters:
The trajectory reports 230 unmatched ModelCallCompleted events as the #1 graph-derived pressure. Day 142's structural guard in src/prompt.rs prevents new agent-code-path orphans, but the harness itself still emits zero-token ModelCallCompleted events (evt-harness-* prefix) without Started counterparts. These inflate the unmatched count and obscure whether any real agent-model lifecycle gaps remain. Closing this harness-internal gap makes the metric trustworthy.

Implementation Notes:
- The evt-harness-* events with zero tokens suggest they're generated by the state recorder's harness infrastructure (possibly RunCompleted or error-recovery paths), not by real model API calls
- Search src/state.rs for where `ModelCallCompleted` events are emitted — look for `record(EventType::ModelCallCompleted` or similar calls
- The fix may be one of:
  a) Add a matching `ModelCallStarted` before each harness-internal `ModelCallCompleted` (preserves event count, pairs them)
  b) Suppress `ModelCallCompleted` emission when no `ModelCallStarted` was recorded for that run (cleaner, but changes event count)
  c) Add a `harness_internal: bool` field to distinguish harness-internal from agent model calls (most flexible, backward-compatible)
- Option (c) is preferred for backward compatibility — it doesn't change existing event counts but allows queries to filter out harness-internal events
- If the harness-internal events represent real model activity (e.g., error-recovery retries), option (a) is correct
- Keep the change scoped to src/state.rs only
